# 日区升级包 Browser Run 生产集成设计规格

**状态：** 已确认，待实施
**日期：** 2026-07-19

## 1. 背景与目标

任天堂日区公开软件搜索 API 会为支持升级的 Switch 2 Edition 根商品返回 `upgrade: 1`，但不会把升级通行证作为独立搜索候选。普通 Worker 请求 My Nintendo Store 详情页时只能取得 JavaScript 排队外壳，无法证明根商品与升级包 URL 的关系。

隔离可行性验证已修复 Wrangler 与 Playwright 的 CDP 兼容问题，并以三个全新远程浏览器实例从同一官方根商品稳定取得唯一升级包 URL。因此本阶段把已验证的 Browser Run 能力接入生产订阅流程，使管理员从默认区选定升级包后，可以自动取得日区候选，同时保留人工官方链接兜底。

本功能只负责日区升级包关系发现和保存前复核。订阅建立后的价格采集仍使用任天堂日区官方价格 API，不允许 Browser Run 进入六小时 Cron、手动刷新、日报或历史查询。

## 2. 已确认决策

1. 采用单 Worker 方案：在现有 `switch-price-monitor` Worker 中增加 Browser Binding，不拆分第二个 Worker。
2. 点击“核验其他地区”后同步等待，前端沿用全局网络加载动画；完成后直接展示日区结果。
3. Browser Run 失败时不在同一请求内自动重试；页面保留重新核验和粘贴日区官方链接。
4. 自动候选在最终保存前必须再次运行 Browser Run，不能复用浏览器提交的旧关系证据。
5. 一次请求共用一个浏览器实例，每个商品使用全新无痕上下文并串行执行；最多允许三个需要日区深度核验的商品。
6. 候选阶段按商品独立降级，单项失败不影响其他商品；最终批量保存继续使用既有 D1 原子写入，任一自动候选复核失败时整批零写入。
7. 人工链接保存时仍先尝试 Browser Run；若浏览器失败，但官方 URL、日区价格 ID、JPY 和在售状态全部验证通过，可按 `manual_link` 保存，不能升级为 `automatic`。

## 3. 生产架构

现有 Worker 增加三个窄职责组件：

1. **日区升级根商品查找器**：只读取任天堂日区官方搜索 API，从外部响应中提取内部根商品证据；`upgrade` 标志不进入前端候选 DTO，也不写入 D1。
2. **Browser Run 关系解析器**：只接收已由根商品查找器验证的日区 Store URL，启动 Playwright 并提取唯一可见的“アップグレードパス”官方链接。
3. **日区升级包关系服务**：协调根商品、浏览器关系和官方价格 API，生成现有订阅向导使用的 `OfficialProductCandidate`，并为最终确认提供同一套二次关系验证。

依赖方向固定为：

```text
官方日区搜索 API
       │ 唯一 upgrade: 1 根商品
       ▼
日区升级包关系服务 ──► Browser Run 关系解析器
       │                     │ 唯一官方升级包 URL
       ▼                     ▼
任天堂日区价格 API ──► 已验证的日区升级包候选
```

`OfficialProductDiscoveryService` 只在“目标地区为 JP、默认区锚点类型为 `upgrade-pack`、普通日区搜索没有同类型候选”时调用关系服务。`JapaneseSubscriptionConfirmationService` 只在保存日区升级包时调用同一服务重新证明关系。其他商品类型和地区保持现有路径。

## 4. 根商品准入

日区根商品必须同时满足以下条件，缺一项即返回受控失败并且不启动浏览器：

- 来自固定的任天堂日区公开软件搜索 API；
- `id` 与 `nsuid` 相同且均为纯数字；
- `sform` 是已准入的下载软件类型；
- `upgrade` 严格等于数字 `1`；
- URL 可按既有规则构造为 `https://store-jp.nintendo.com/item/software/D数字/`；
- 根商品类型是完整游戏，标题明确包含 `Nintendo Switch 2 Edition`；
- 与默认区升级包锚点具有相同的规范化发行商和拉丁系列标记；
- 去重后恰好一个根商品。

根商品为零或多个时不能按搜索顺序任选一个，也不能扩大查询、翻译标题、枚举猜测 ID 或访问第三方站点。

## 5. Browser Run 关系校验

### 5.1 URL 与 DOM 约束

浏览器只允许导航到根商品查找器返回的日区官方 URL。页面渲染后只读取可见文字包含 `アップグレードパス` 的链接，并对每个目标执行以下校验：

- 协议必须是 `https:`；
- 主机必须精确为 `store-jp.nintendo.com`；
- 不允许端口、用户名、密码、查询参数或片段；
- 路径必须匹配 `^/item/software/D[0-9]+/?$`；
- 接受零个或一个末尾斜杠后，统一规范化为带斜杠 URL；
- 去重后必须恰好一个结果；零个、多个、隐藏链接或无效 URL 均失败。

Browser Run 不点击验证码、不登录 Nintendo Account、不尝试绕过排队或机器人控制，也不执行加入购物车、购买或账号操作。

### 5.2 候选身份与价格

自动日区候选由服务端从三项官方证据构造：

- `productUrl` 来自 Browser Run 取得的唯一关系链接；
- `canonicalTitle` 使用已验证日区根商品标题追加固定官方关系名称 `アップグレードパス`；
- `publisher` 来自同一日区官方根商品，而不是默认区锚点；
- `productType` 固定为 `upgrade-pack`，`regionCode` 固定为 `JP`，`currency` 固定为 `JPY`；
- 当前价、原价和折扣状态来自任天堂日区官方价格响应；Browser 页面文字不作为价格数据来源。

升级包 URL 中的数字 ID 必须由价格 API 再次证明为 `JP / JPY / onsale / 同一 title_id`。候选卡只在官方响应同时提供有效常规价和更低折扣价时显示划线原价与折扣力度；价格结构异常时关系候选不得伪装成免费商品。

## 6. 最终确认与人工兜底

### 6.1 自动候选

`automatic` 日区升级包在保存前必须重新执行完整链路：唯一根商品、Browser Run 唯一关系 URL、同一升级包 URL 和日区价格 API。浏览器提交的标题、发行商、价格、候选状态或旧链接都不能替代本次官方结果。任一环节失败时，确认服务返回安全中文错误，既有 D1 原子批次不得执行。

### 6.2 人工链接

`manual_link` 仍先尝试完整 Browser Run 关系复核。若浏览器启动、导航、页面结构或额度暂时失败，可在以下条件全部成立时保存：

- 管理员提交的是精确日区官方升级包 URL；
- URL 中的 ID 由日区价格 API 证明为 JP、JPY、在售和同一 ID；
- 默认区锚点类型明确为 `upgrade-pack`；
- 匹配来源保持 `manual_link`，不得写成 `automatic`。

人工兜底是管理员显式承担关系确认责任的较弱路径。URL 或价格验证失败不属于 Browser Run 暂时不可用，不能通过人工来源绕过。

## 7. 请求、并发与资源生命周期

- 每个 `/api/products/resolve-regions` 或 `/api/products/confirm-subscriptions` 请求最多启动一个浏览器实例。
- 每个待核验商品创建全新无痕 Browser Context；上下文不得复用 Cookie、缓存、localStorage、IndexedDB 或排队状态。
- 商品按稳定输入顺序串行执行，不并行创建浏览器或页面。
- 单商品导航和关系提取最多 30 秒；超时后不自动重试。
- 一批超过三个需要日区深度核验的商品时，在浏览器启动前返回明确的 `422` 提示，要求管理员分批处理；不得只处理前三个或静默忽略其余商品。
- 每个页面和上下文在单项 `finally` 中关闭；浏览器在请求级 `finally` 中关闭。关闭失败只能记录安全状态，不能覆盖原业务结果或泄漏异常正文。
- 不跨请求保持 Browser Session，不使用 `keep_alive`，不把会话 ID 写入 D1、日志或响应。

Cloudflare Workers Free 当前允许每天 10 分钟 Browser Run、每账号 3 个并发浏览器和每 20 秒新建 1 个实例。连续快速操作若触发额度或频率限制，按本规格降级为可重新核验的安全失败，不在请求内等待后重试。

## 8. API 与前端行为

现有公开 API 路径保持不变，不新增公开 Browser Run 或诊断端点：

- `POST /api/products/resolve-regions` 在满足日区升级包回退条件时同步执行 Browser Run；成功仍返回 `automatic` 候选。
- 单项浏览器失败返回该商品/日区的人工链接状态和安全中文说明，其他商品的地区结果继续返回。
- 超过三个深度核验商品返回 `422`，不启动浏览器。
- `POST /api/products/confirm-subscriptions` 对自动日区升级包执行二次关系验证；失败保持现有批量零写入语义。

前端继续使用共享请求计数器驱动全局加载动画。日区失败项显示“日区自动核验暂不可用”，并提供“重新核验”和“粘贴日区官方链接”；不得显示底层错误、HTML、响应正文或 Cloudflare 会话信息。

## 9. 配置与依赖

生产实现固定采用以下已验证组合：

- `@cloudflare/playwright`：`1.3.0`；
- `wrangler`：`4.112.0`；
- `@cloudflare/workers-types`：`5.20260714.1`；
- `compatibility_date`：保留 `2026-07-16`；
- `compatibility_flags`：增加 `nodejs_compat`；
- Browser Binding：`BROWSER`。

依赖与锁文件必须一起更新。`Env` 只增加 Browser Binding 类型，不增加 API Token、账号凭据或 Secret。Browser Run 直接通过绑定启动，不允许把 Cloudflare API Token 写入代码、D1、前端或普通日志。

## 10. 日志与数据保留

本功能不新增 D1 表、缓存、Cron 或持久关系记录。结构化日志只允许包含事件名、归一化阶段、成功/失败状态、耗时和本批数量；错误名称和消息必须按既有诊断脱敏规则收窄。

禁止记录或返回：完整 HTML、页面正文、Cookie、localStorage、IndexedDB、队列令牌、请求头、响应头、截图、Trace、HAR、浏览器 Session ID、异常堆栈、密码、会话令牌和 Telegram 凭据。

## 11. 测试与发布门禁

### 11.1 自动化测试

1. 根商品解析测试覆盖 `upgrade: 1`、ID/NSUID、下载类型、系列、发行商、Switch 2 Edition、零个和多个根。
2. URL 与 DOM 纯函数测试覆盖可见文字、唯一性、末尾斜杠规范化、错误主机/协议/路径、查询参数、片段、端口、凭据和隐藏链接。
3. 浏览器协调测试覆盖单浏览器、最多三个全新上下文、串行顺序、30 秒超时、所有成功与异常路径关闭资源，以及零自动重试。
4. 关系服务测试覆盖官方价格 ID、JP/JPY/onsale、同一 ID、当前价/原价、自动候选字段和单项失败独立降级。
5. 最终确认测试覆盖自动候选二次 Browser Run、人工链接浏览器失败后的价格兜底、人工来源不可升级，以及批量零部分写入。
6. API 与 DOM 测试覆盖管理员认证、三个商品上限、全局加载动画、安全中文提示、重新核验和人工链接入口。
7. 完整回归必须通过 Worker 测试、DOM 测试、TypeScript 类型检查、生产构建、中文注释一致性、敏感信息扫描和 `git diff --check`。

### 11.2 发布验收

实现完成后先在本地模拟 Browser Binding 验证全部失败路径，再使用远程预览对已知 Overcooked! 2 正向样本执行只读核验。生产部署必须单独取得管理员确认，部署时按既有临时版本规则从 V0.0.12 升至 V0.0.13。

生产首次验收只调用已认证的只读地区核验接口，确认返回日区升级包官方 URL、JPY 当前价和 `automatic` 状态；不得自动创建、修改或删除订阅。真实最终写入需管理员在页面明确确认。

## 12. 非目标

- 不把 Browser Run 用于日常价格采集、汇率、Telegram、仪表盘或历史查询。
- 不为其他地区或普通游戏、组合商品、DLC 启用浏览器解析。
- 不引入第三方价格站、搜索引擎、静态 ID 对照表或 Nintendo Account。
- 不新增缓存、数据库迁移、后台队列、轮询接口、独立 Browser Worker 或自动重试。
- 不在本规格中恢复手动价格刷新的 15 分钟冷却，也不实施 Telegram 设置。

## 13. 文档自检

- 本规格所有章节均已完整定义，且不存在未决数据来源。
- 同步等待、无自动重试、最多三个商品、候选独立降级和最终原子写入互不冲突。
- 自动候选与人工链接的信任边界明确：自动必须二次关系证明，人工只能在浏览器暂时失败且官方 URL/价格均有效时兜底。
- Browser Run 仅进入订阅发现和确认，不会改变既有六小时价格采集、日报、D1 数据模型或生产版本规则。
- 配置、依赖、资源关闭、日志禁区、测试和只读生产验收均具有明确可验证条件。

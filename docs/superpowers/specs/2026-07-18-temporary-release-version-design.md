# 临时发布版本号设计规格

**状态：** 已实施，生产部署待管理员授权
**日期：** 2026-07-18

## 1. 目标

在单一生产环境中，为管理员提供可见、可追溯的前端发布版本号，以便快速区分当前打开的页面是否已经包含最近一次部署。该机制是过渡方案，未来可替换为 Git 标签、CI 构建编号或其他正式发布系统。

## 2. 已确认行为

1. 左侧导航底部始终显示 `V x.y.z`，首个受此规则管理的版本为 `V 0.0.1`。
2. `package.json` 的 `version` 是唯一版本来源；页面不得维护第二份硬编码版本字符串。
3. `npm run build`、`npm run dev` 和测试均不改写版本，避免本地调试、构建失败或测试执行造成版本漂移。
4. `npm run deploy` 在构建前自动将补丁版本递增一次，例如 `0.0.1` 递增为 `0.0.2`，再把该版本注入静态前端并部署生产 Worker。
5. 自动递增会修改受 Git 管理的版本清单；发布前仍须经过管理员明确确认，发布完成后按项目规则将版本变更与功能变更一起提交并推送。

## 3. 实现边界

- 新增单一构建时版本模块，由 Vite 在构建时读取 `package.json` 的版本并导出给 React 外壳；浏览器不读取文件系统，也不请求额外 API。
- 将现有起始 `package.json` 版本调整为 `0.0.1`，使页面首次展示与已确认的 `V 0.0.1` 一致；同步更新 lockfile 的根包版本，保持依赖元数据一致。
- 使用 npm 内置的 `version patch --no-git-tag-version` 完成递增，不创建自动 Git tag。该命令只在部署脚本内调用，且必须在构建之前执行。
- 版本号仅用于发布识别，不参与认证、价格计算、D1 迁移、缓存安全策略或 Telegram 消息。

## 4. 验收标准

1. 在 `package.json` 为 `0.0.1` 时，认证后左侧导航显示 `V 0.0.1`。
2. 应用外壳的 DOM 测试覆盖该文本，防止后续布局重构遗漏版本标识。
3. 普通构建前后 `package.json` 版本不变。
4. 部署脚本的命令顺序先执行 `npm version patch --no-git-tag-version`，再构建和部署；每次成功启动生产部署只递增一次补丁号。
5. 页面不暴露密钥、构建环境变量或 Git 提交内容。

## 5. 范围外

- 不创建测试/staging 环境。
- 不引入 CI/CD、Git tags、自动发布记录或回滚系统。
- 不把版本号保存到 D1，也不要求管理员在页面中手动编辑版本。

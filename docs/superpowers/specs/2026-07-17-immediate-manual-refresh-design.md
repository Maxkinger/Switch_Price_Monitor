# 立即手动刷新设计规格

## 1. 状态与范围

| 项目 | 内容 |
| --- | --- |
| 状态 | 已确认，待实施 |
| 日期 | 2026-07-17 |
| 目标 | 管理员点击“立即刷新”后在同一受保护请求中完成一次真实官方采集，并返回本次结果。 |

本规格只调整手动刷新的执行时机。六小时自动采集、历史保留、官方优先来源链、汇率、降价规则与 Telegram 配置边界保持不变；不接入第三方价格站或 Telegram 凭据。

## 2. 已确认行为

1. `POST /api/refresh` 必须先通过管理员会话，并使用服务端时间执行 15 分钟原子冷却。
2. 未处于冷却时，接口立即调用与六小时 Cron 完全相同的 `LiveCollectionRunner`，等待本轮完成后返回 `attempted`、`collected`、`stale` 与执行时间。
3. 成功响应表示采集轮次已结束，不把“已接受”“已排队”误写成价格已经更新；`stale` 表示该地区未取得可验证价格，不创建伪造快照。
4. 前端收到结果后显示成功/待确认数量，并重新读取仪表盘，使新快照或过期状态立刻可见。
5. 六小时 Cron 保持固定 `0 */6 * * *`，独立执行历史保留和完整采集；不再读取、认领或消费手动刷新队列。

## 3. 架构与数据流

```text
管理员点击立即刷新
        │
        ▼
POST /api/refresh（会话 + 15 分钟原子冷却）
        │
        ▼
LiveCollectionRunner（官方来源、汇率、快照、健康状态）
        │
        ▼
返回本轮统计 → 仪表盘重新读取

六小时 Cron ──► 保留清理 + 同一 LiveCollectionRunner
```

- 刷新路由只负责编排认证、冷却和运行器，不复制来源、汇率、快照或通知规则。
- 运行器沿用逐地区串行采集语义：一个地区失败不会阻断其它地区；外部页面、Cookie 和原始响应不写入接口响应或日志。
- 冷却记录只保存最近一次手动刷新时间，不再有 `queued` 或 `running` 状态。`0006` 迁移会重建该单行表且不复制旧记录：这是有意取消旧语义下所有等待 Cron 的请求，避免把既有 `queued` 行误认为已完成的立即采集；该表不承载价格、订阅或认证业务数据。

## 4. 错误与安全边界

- `401`：未认证，不启动任何采集。
- `429`：仍在冷却，返回服务端计算的下一次允许时间；不能由浏览器时间绕过。
- 单地区来源不可用：记录既有安全失败/过期结果，返回计入 `stale`，但 HTTP 请求整体可完成。
- 运行器意外失败：返回安全 `500` 摘要，不暴露 D1、供应商 URL、Cookie、响应正文或 Telegram 信息。
- 同一时刻的 Cron 与手动请求都使用相同的运行器；实现必须避免读取旧队列形成第二次采集路径。

## 5. 验收

- 首次手动请求在无需等待 Cron 的情况下返回本轮统计；十五分钟内的第二次请求返回 `429`。
- 六小时 Cron 仍可独立采集，且不依赖手动刷新状态。
- 成功、部分失败和全量无可用价格时，快照与健康状态遵循既有来源规则。
- 前端在成功后显示本轮结果并重新读取仪表盘；未认证不会保留管理员数据。
- 生产验收在管理员明确允许后执行一次真实手动采集，不输入或展示 Telegram 凭据。

# 美食家版跨区官方组合商品识别设计规格

**状态：** 已确认香港官方关联商品展开方案，待实施
**日期：** 2026-07-18

## 1. 背景与问题证据

管理员在美区官方候选中选择 `Overcooked! 2 - Gourmet Edition` 后，现有向导无法为日区、港区完成地区核验。

2026-07-18 的只读官方验证确认：

- 日区公开软件搜索 API 返回 `Overcooked® 2 - オーバークック２：真の食通エディション`，ID 为 `70070000010202`，形态为 `DL_DLC`，官方价格 API 返回常规价 ¥4,900、当前促销价 ¥1,225。
- 港区官方 eShop 存在 `Overcooked! 2 - Gourmet Edition`，地址为 `https://ec.nintendo.com/HK/zh/bundles/70070000010913`，页面明确标记为组合商品，发行商为 Team17，公开售价 HKD 354。

本轮工作开始前的实现只接受日区以 `*_DL` 结尾的下载形态，并把港区检索限定在普通软件索引和 `/titles/` 路径。因此两条已验证的官方组合商品都曾在解析或链接准入阶段被排除；这不是商品不存在，也不应由管理员手工填入价格 ID 绕过。

后续生产只读诊断又确认：港区 Magento 商城搜索在普通网络环境可返回目标商品，但从 Cloudflare Worker 请求时固定返回 HTTP 403；同一次 Worker 请求中的香港普通官网搜索返回 HTTP 200。解析器用真实商城 HTML 离线重放时能够正确读取 `bundles/70070000010913`，所以根因是商城入口对 Worker 出口的访问限制，而不是标题匹配、HTML 解析或 30 秒超时预算。

进一步只读验证发现，香港普通官网搜索 `Overcooked 2` 可返回本体 `70010000033098` 与 Switch 2 Edition `70010000106253`。本体的 `ec.nintendo.com` 官方详情数据公开包含：

- `includedBundleItems`：包含 `Overcooked! 2 - Gourmet Edition`，NSUID 为 `70070000010913`；
- `dlcItems`：包含本体关联的官方新增内容；
- `upgradeInfo`：包含 Switch 2 Edition 升级通行证，NSUID 为 `70050000065163`。

因此港区可以从可访问的普通官网本体候选出发，只展开一层任天堂官方关联商品，再对目标组合商品执行现有官方详情与价格验证，无需继续请求受限的 Magento 搜索页。

## 2. 目标与非目标

### 目标

1. 在新建订阅的跨区核验中，将美区美食家版与唯一、官方、同类型的日区/港区等价组合商品自动匹配。
2. 日区和港区都沿用各自任天堂公开价格 API，先从官方商品 URL 提取本区 ID，再二次验证价格与 ID。
3. 维持现有安全边界：原版、Switch 2 Edition、升级包、DLC、季票或存在歧义的组合商品不得被自动映射到美食家版。

### 非目标

- 不把同一系列、同一发行商或同一主标题本身视为等价证明。
- 不使用第三方搜索、翻译、AI、Nintendo Account、Cookie 或浏览器会话。
- 不修改已存在订阅、地区映射、价格历史或通知；本次只改变后续发现、确认和采集能力。
- 不建立通用的任意日文/中文版本词翻译系统；只接受受控、可审计的“美食家版”标记集合。

## 3. 方案比较与选择

1. **普通官网搜索后展开官方关联商品（采用）：** 普通官网负责发现本体 NSUID，`ec.nintendo.com` 本体详情负责提供组合包、DLC 与升级关系；随后逐个重读关联商品详情并执行既有身份和价格验证。数据仍全部来自任天堂官方，同时绕开 Magento 403。
2. **增加非 Cloudflare 抓取服务：** 使用 VPS 或浏览器服务请求 Magento 商城，再把结果传回 Worker。它能复用现有解析器，但会新增服务器、认证、可用性和隐私边界；当前官方关联数据已经足够，不引入该架构。
3. **仅保留人工官方链接：** 实现简单，但管理员仍需逐区查找并粘贴商品，不能满足自动匹配已存在官方关系的组合包和升级包；仅作为关系数据缺失或不可验证时的安全兜底。

## 4. 设计

### 4.1 日区官方搜索与类型

- 日区搜索适配器增加经过本次实测的 `DL_DLC` 下载组合形态；实体卡带、带 `-2` 的实体条目和未知形态仍拒绝。
- `DL_DLC` 不表示普通 DLC：对于该官方形态，候选被标记为 `bundle`，以保持与美区 `BUNDLE` 候选相同的领域类型。
- 日区数字 ID 继续映射为 `https://store-jp.nintendo.com/item/software/D{ID}/`；最终价格验证使用既有 JP 公开价格 API，不依赖可能出现排队页的商品 HTML。
- 当英文锚点查询没有任何同类型候选时，系统可执行**至多一次**日区二次检索：只能从首轮已验证的日区官方结果中提取唯一片假名系列别名，且该结果须与锚点同发行商、同拉丁系列标记。别名缺失、多个别名、首轮已有同类型候选或二次响应不可用时，不继续扩展查询，保持人工链接路径。
- 日区最终订阅复核以待保存候选的官方日区标题检索，并继续要求响应出现相同精确 Store URL 与官方价格 ID；这解决英文美食家版标题无法重新索引日文本地化组合商品的问题，但不信任浏览器标题本身。

### 4.2 港区官方组合商品

- 港区名称发现只请求可由 Worker 稳定访问的 `https://www.nintendo.com/hk/search`；不再把 Magento 商城搜索作为自动发现依赖，也不使用第三方索引、搜索引擎缓存或账号会话。
- 当原始标题没有同类型港区候选，且锚点标题含有已受控的版本标记时，系统可移除该版本后缀生成一次基础标题查询。例如 `Overcooked! 2 - Gourmet Edition` 只能生成 `Overcooked! 2`；不能翻译未知词、枚举任意系列名或进行无限次搜索。
- 基础标题查询返回的每个 `titles/{数字 ID}` 候选必须先通过香港官方 URL、地区、语言和数字 NSUID 校验。系统最多读取前五个已验证本体详情；超过边界不继续扩大请求范围，而是回退人工官方链接。
- 每个本体详情只展开一层公开关系：`includedBundleItems` 映射为 `bundle`，`dlcItems.items` 映射为 `dlc`，`upgradeInfo[].upgradeDlcItem` 映射为 `upgrade_pack`。不递归读取关联商品的更多关系，避免循环、重复和不受控请求放大。
- 单个本体详情若产生超过五十个去重后的关联商品，视为外部结构异常并放弃自动展开；不能截断后再依据顺序自动选择，以免遗漏同名候选破坏唯一性。
- 关系对象只提供待验证的 NSUID、标题、类型和封面。系统必须据此生成精确 `ec.nintendo.com/HK/zh/{titles|aocs|bundles}/{数字 ID}` 地址，并重读该关联商品自己的官方详情；发行商不得从本体复制，避免不同发行商的组合商品借用本体身份信号。
- 港区 URL 白名单和商品页解析器增加精确路径 `/HK/zh/bundles/{数字 ID}`。该路径只允许 HTTPS、`ec.nintendo.com`、繁体中文港区和纯数字 ID。
- 港区价格 ID 提取器同样只增加 `bundles` 资源类型；仍拒绝查询参数、片段、其他语言、其他地区、其他主机和非数字路径。
- 组合商品页面公开元数据的标题、发行商和封面可用于身份验证；价格必须继续由官方价格 API 读取，不能从页面可见文本猜测。
- 实测 `bundles` 页面不包含普通商品页面的 `search.*` 元标签；解析器只读取 Next RSC 中 `BundleItem` 的 `formalName`、`publisher.name` 和封面字段，并要求其中 `nsUid` 等于请求 URL 的数字 ID。页面内推荐商品、缺失发行商或结构变化均安全拒绝。
- 关联商品完成详情复核后，仍须通过同类型、同发行商、相同拉丁主标题、相同受控版本标记和唯一候选约束，才能自动确认；官方“关联”只负责发现候选，本身不能替代跨区等价证明。

### 4.3 等价组合商品的自动匹配

在既有严格标题匹配之后，新增仅适用于 `bundle` 的受控版本标记：

| 地区标题标记 | 规范化版本标记 |
| --- | --- |
| `Gourmet Edition` | `gourmet-edition` |
| `真の食通エディション` | `gourmet-edition` |

自动确认需同时满足：

1. 双方均为本区官方 URL，且 `productType === bundle`；
2. 双方发行商存在且经现有 Unicode/大小写规范化后相同；
3. 双方可提取相同的拉丁主标题标记，例如 `overcooked2`；
4. 双方均提取到同一受控美食家版本标记；
5. 本次该地区官方结果中恰好有一个候选满足以上条件。

任何条件不满足时保持现有人工候选或官方链接入口。版本标记仅作为与主标题、发行商和类型组合的证据，不能独立跨区识别商品。

### 4.4 错误与兼容边界

- 日区/港区官方搜索网络或结构失败时保留“官方搜索不可用，请粘贴官方链接”的现有安全行为。
- 港区普通搜索成功但基础标题推导、关系解析或关联商品详情复核失败时，不把本体错误展示为组合商品候选；目标类型没有可验证候选时回退人工官方链接。
- Magento 商城 HTTP 403、匿名 REST 标准索引缺失目标商品以及 Savanna GraphQL 未提供经验证的公开名称搜索，都不能触发非官方回退或放宽 URL 校验。
- 港区普通软件搜索与既有 `titles/aocs` 商品不改变；本次只新增 `bundles`。
- 既有 Switch 2 Edition 本体的跨语言高置信度规则不改变；其版本标记与美食家版标记彼此不等价。
- 最终订阅确认仍重读官方来源、重算自动身份唯一性，并通过官方价格 API 验证 ID；浏览器提交的标题、类型或 `automatic` 状态不构成信任依据。

## 5. 测试与验收

测试先行，至少覆盖：

- 日区 `DL_DLC` 美食家版被解析为 `bundle`，生成正确 Store URL 和日元金额；实体卡带及未知形态仍被拒绝。
- 港区官方关联商品展开、`/bundles/` 商品页验证、价格 ID 提取和官方价格 API 预览均成功。
- 港区原始美食家版查询无同类型候选时，只执行一次 `Overcooked! 2` 基础标题搜索；从本体 `includedBundleItems` 展开并复核 `70070000010913` 后自动匹配。
- 本体 `upgradeInfo` 可生成升级通行证候选，普通 `dlcItems` 只能生成 DLC 候选；三种类型不能互相替代。
- 关系超过一层、根本体超过五个、单页关联商品超过五十个、NSUID/类型/路径不一致或关联详情缺少发行商时均不能自动匹配。
- 港区自动发现流程不再请求 Magento 商城搜索，临时网络诊断字段在新流程稳定后移除。
- 美区、日区、港区三种美食家版标题在唯一官方候选时自动匹配。
- 原版、Switch 2 Edition、季票、升级包或多个同等候选不能自动匹配。
- 回归既有 JP `BEE_DL`、HK `titles/aocs`、美/墨/巴候选搜索和最终确认。

生产受控验收仅在管理员主动选择美区美食家版并点击“核验其他地区”后检查日区、港区自动确认与官方价格来源预览；除非管理员另行点击最终“确认订阅”，不得创建或修改订阅数据。

## 6. 文档自检

- 未包含 TBD、TODO 或未决价格来源。
- 类型、URL、ID 和价格读取边界均限定到已验证的任天堂官方资源。
- 自动匹配条件与失败回退一致，未把人工候选或同系列关系误写为自动确认。

## 7. 实施记录

- 2026-07-18：日区搜索仅为经过验证的 `DL_DLC` 形态返回 `bundle`；既有下载本体和升级包仍由官方标题分类，实体和未知形态继续拒绝。
- 2026-07-18：港区搜索新增官方 eShop 组合商品结果；`bundles/{ID}` 与既有 `titles/aocs` 分别受到精确主机、地区、语言、资源类型、数字 ID、无查询/片段的校验。组合搜索缺发行商时，匹配器只能重读同 URL 官方详情页补齐。
- 2026-07-18：高置信度本地化规则新增受控 `Gourmet Edition` / `真の食通エディション` 标记；日区与港区美食家版均须同时满足同 `bundle`、同发行商、相同拉丁主标题、同版本标记和唯一候选才自动确认。
- 2026-07-18：根据生产只读验收，日区增加从唯一官方系列结果推导的单次日文别名检索，并将最终复核搜索词改为候选官方日区标题；港区组合页改为校验 RSC `BundleItem.nsUid`、标题和发行商，未放宽 URL、类型、价格 API 或唯一性边界。
- 2026-07-18：第二次生产只读验收确认港区商城官方搜索页可返回组合商品，但其约 25 秒响应超过统一 12 秒预算；因此仅为该固定官方 bundles 搜索入口设置 30 秒上限，并以回归测试锁定其他搜索预算不变。
- 2026-07-18：V 0.0.8 仍在生产回退手动链接后，实测 HTML 重放确认商城搜索和 `BundleItem` 解析均正确，故将临时诊断从不可读取的 Worker 控制台改为已认证管理员本次跨区核验响应。该摘要仅包含固定入口类型、结果、HTTP 状态和耗时；不含查询词、URL、Cookie、响应正文、商品标题或可持久化数据，并在修复后移除。
- 2026-07-18：V 0.0.10 管理员诊断确认 Magento 商城搜索从 Worker 返回 HTTP 403，而香港普通官网搜索返回 HTTP 200。只读研究进一步验证《Overcooked! 2》本体详情的 `includedBundleItems`、`dlcItems` 与 `upgradeInfo` 已公开目标组合包、DLC 和升级通行证 NSUID；设计因此改为“一次基础标题搜索 + 一层官方关系展开 + 关联商品详情复核”，并停止把 Magento 商城作为自动发现依赖。

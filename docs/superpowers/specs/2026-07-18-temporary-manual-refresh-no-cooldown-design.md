# 临时取消手动刷新冷却设计规格

## 状态

已部署并完成生产香港区验证。确认、本地质量门禁与生产验证日期：2026-07-18。

## 目标

在香港区生产验证期间，临时取消管理员手动刷新的 15 分钟服务器冷却，使每一次已认证点击都立即执行同一套价格采集流程。

## 范围与边界

1. 仅取消 `POST /api/refresh` 的手动刷新冷却；六小时 Cron 的独立价格采集节奏不改变。
2. 手动刷新仍必须经过现有管理员会话认证；不新增公开端点、不放宽登录或会话安全边界。
3. `manual_refresh_requests` 继续仅保存单行 `requested_at`，用于显示最近刷新时刻和故障诊断；不保存 IP、浏览器、会话、商品或外部响应。
4. 每次请求都无条件 UPSERT 该时间戳，并返回 `accepted: true`；`nextAllowedAt` 与本次 `requestedAt` 相同，避免客户端继续显示不存在的倒计时。
5. 不新增 D1 迁移、不清空既有冷却记录、不改变价格快照、采集日志、汇率、通知或订阅数据。
6. 这是临时开发验证规则。恢复限流时必须另行确认，并重新启用原有的 15 分钟条件 UPSERT、429 响应和并发限流测试；不得默默保留无限频率刷新行为。

## 数据流

```text
已认证管理员点击“立即刷新”
  -> 无条件写入最近刷新时间
  -> accepted: true，nextAllowedAt = requestedAt
  -> 当前请求同步执行采集器
  -> 返回本轮统计并由页面重读最新地区价格
```

## 错误与并发规则

- 采集器失败仍返回既有的安全 500 错误，不能回显外部 URL、响应正文、SQL 或堆栈。
- 不存在 429 冷却错误；并发点击各自可触发一轮采集，这是本期“取消限制”的明确后果。
- 最近刷新时间只记录最后一次成功进入刷新流程的请求，不能当作排队状态或 Cron 任务。

## 验收标准

- 连续两次管理员手动刷新都返回 200，且采集器收到两次调用。
- 两次请求后 `manual_refresh_requests` 只保留一行，时间为最近一次请求时间。
- 未登录请求继续被认证层拒绝；采集器异常继续返回安全错误。
- Worker、页面、类型检查和生产构建通过；生产验证中可在不等待 15 分钟的情况下重试香港价格采集。
- 生产 V0.0.5 已验证立即刷新完成五区采集且没有返回冷却错误；香港区成功写入官方价格。

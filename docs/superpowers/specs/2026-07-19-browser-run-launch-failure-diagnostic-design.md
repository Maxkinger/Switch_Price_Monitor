# Browser Run 启动失败受控诊断设计规格

**状态：** 已完成；启动根因已修复，三次任天堂复验通过
**日期：** 2026-07-19

## 1. 背景与问题定义

日区升级包 Browser Run 隔离探针已连续执行三次，三次均在 `@cloudflare/playwright` 的 `launch()` 阶段返回 `browser-launch-failed`。现有实现为遵守最小输出原则，在捕获启动异常时丢弃了错误名称、错误码和简短错误信息，因此已有证据只能证明“浏览器没有启动”，不能区分以下根因：

- 临时探针代码、依赖或兼容配置错误；
- 本地开发浏览器环境错误；
- 远程 Browser Binding 未正确注入；
- Cloudflare 账户权限、免费额度、实例频率或并发限制；
- Cloudflare Browser Run 平台暂时异常。

在查明根因前，不得把启动失败解释为任天堂排队页或反自动化限制，也不得直接把 Browser Run 接入生产商品发现流程。

## 2. 目标与非目标

### 2.1 目标

1. 在既有临时探针中增加经过脱敏的启动诊断能力，保留足以判断失败阶段和平台错误类别的非敏感证据。
2. 使用相同代码依次运行本地浏览器和远程 Browser Binding，形成只改变执行环境的 A/B 对照。
3. 浏览器基础设施通过后，依次验证普通测试页导航和任天堂日区详情页，避免跨阶段归因。
4. 找到单一、可复现的根因后，只修改与该根因对应的最小代码或配置。
5. 远程浏览器和任天堂页面关系提取均恢复后，重新执行原设计要求的三次独立验证。

### 2.2 非目标

- 不修改或部署生产 Worker，不改动生产 D1、Cron、Static Assets、Secrets、订阅或价格历史。
- 不在本阶段设计 Browser Run 的生产调用频率、缓存、费用、监控或回退策略。
- 不使用第三方站点、搜索引擎缓存、Nintendo Account、登录会话或验证码绕过。
- 不把本地浏览器成功视为远程 Browser Binding 成功，也不把远程启动成功视为任天堂页面可达。
- 不通过重复重试规避 Cloudflare 额度、频率或并发限制。

## 3. 方案比较与决策

### 3.1 方案 A：本地与远程受控 A/B 诊断（采用）

在同一份隔离探针中增加 `launch-only` 诊断路径，先以本地 `wrangler dev` 启动浏览器，再以远程 Browser Binding 启动浏览器。两个环境使用相同的源代码、依赖版本和阶段定义，只改变浏览器执行环境。该方案能把“代码与本地运行问题”和“远程账户、绑定或平台问题”分开，且不创建持久云端资源。

### 3.2 方案 B：部署临时诊断 Worker（暂不采用）

一次性部署更接近生产 Worker 运行环境，也便于使用持续日志，但会创建持久外部状态并需要单独部署、清理和授权。只有远程开发模式无法提供有效诊断、且管理员另行明确批准时，才重新评估本方案。

### 3.3 方案 C：改用 REST Quick Actions 或 CDP（暂不采用）

REST Quick Actions 或 CDP 可以绕开当前 Playwright `launch()` 路径，但会同时改变调用方式、权限模型和运行接口，不适合用于定位当前路径的单一根因。只有确认 Playwright 路径本身不适用后，才作为新的技术方案单独设计。

## 4. 隔离诊断架构

诊断继续位于 `/tmp/switch-price-monitor-jp-browser-probe`，不得复制到生产 `src/` 或生产 Wrangler 配置。临时探针增加三个职责单一的模块：

- `diagnostic-domain.ts`：把未知异常转换成严格白名单诊断结果，执行错误名称和错误码约束、控制字符清除、敏感片段脱敏与长度限制。
- `launch-diagnostic.ts`：按“绑定检查 → 浏览器启动 → 创建上下文 → 创建页面”顺序执行 `launch-only`，记录失败阶段，并在所有返回路径关闭已创建资源。
- `diagnostic-http.ts`：提供只接受本地触发的诊断入口，拒绝无关方法和路径，并为 JSON 响应设置 `Cache-Control: no-store`。

原有任天堂关系探针保持独立。`launch-only` 不导航到任何网页；只有本地和远程启动诊断均通过后，才执行导航验证。

## 5. 诊断数据结构与安全边界

### 5.1 允许字段

单次诊断只允许返回：

- `status`：`success` 或 `failure`；
- `executionTarget`：`local` 或 `remote`；
- `stage`：`binding-check`、`browser-launch`、`context-create`、`page-create`、`page-navigation` 或 `complete`；其中 `page-navigation` 只用于固定普通测试页，不允许由请求提供目标地址；
- `errorName`：受长度和字符集约束的错误名称，仅在失败时存在；
- `errorCode`：受长度和字符集约束的短错误码，仅在可安全提取时存在；
- `errorMessage`：脱敏、去控制字符并截断后的简短错误信息，仅在失败时存在；
- `elapsedMs`：非负整数执行耗时。

### 5.2 禁止字段与脱敏规则

诊断不得输出或保存完整堆栈、`cause` 对象、页面 HTML、响应正文、Cookie、存储状态、请求头、响应头、账号标识、API Token、排队令牌或截图。

`errorMessage` 必须在纯函数中完成以下处理后才能离开 Worker：

1. 删除换行和其他控制字符，防止日志注入；
2. 对 URL 查询参数、Bearer 值、常见 token/secret/password 键值和长随机串进行统一替换；
3. 将连续空白收敛为单个空格；
4. 限制最大长度，禁止把平台响应正文通过错误信息带出；
5. 对非 `Error` 异常使用固定兜底文本，不序列化任意对象。

临时结果文件只保存上述白名单字段，不进入 Git。找到根因后，详细诊断入口从临时探针移除或恢复为原有归一化状态，不并入生产日志。

## 6. 分阶段数据流

### 6.1 第一阶段：本地 `launch-only`

使用本地 `wrangler dev` 运行相同探针，仅启动浏览器、创建上下文、创建空白页面并按逆序关闭资源。失败表示应优先检查本地 Chrome、依赖版本、Wrangler 或兼容配置；在本地失败解决前不得运行远程诊断。

### 6.2 第二阶段：远程 `launch-only`

本地成功后，使用远程 Browser Binding 运行同一诊断。若仅远程失败，则根据脱敏错误证据检查绑定注入、账户权限、额度、频率、并发或平台状态；不得修改页面导航逻辑。

### 6.3 第三阶段：普通测试页导航

远程启动成功后，使用独立验证访问无业务依赖的普通测试页。该阶段只判断远程会话能否完成基础网络导航，不提取任天堂数据，也不改变升级包关系准入结论。

### 6.4 第四阶段：任天堂日区详情页

普通测试页成功后，恢复原探针对指定日区官方详情页的只读访问。若浏览器已成功启动但页面仍未出现唯一合规升级链接，才可归类为任天堂排队页、页面结构或自动化访问限制。

### 6.5 第五阶段：三次独立验证

任天堂单次验证成功后，按原规格连续执行三次全新远程浏览器实例验证。三次均须在规定时间内返回同一个官方升级包 URL；任意一次失败都不得进入生产集成设计。

## 7. 判断矩阵与修复约束

| 本地启动 | 远程启动 | 普通导航 | 任天堂提取 | 允许结论 |
| --- | --- | --- | --- | --- |
| 失败 | 未执行 | 未执行 | 未执行 | 本地运行、依赖或配置问题 |
| 成功 | 失败 | 未执行 | 未执行 | 远程绑定、账户、额度、频率或平台问题；以错误证据细分 |
| 成功 | 成功 | 失败 | 未执行 | 远程浏览器网络导航或会话问题 |
| 成功 | 成功 | 成功 | 失败 | 任天堂页面可达性、结构或自动化访问限制 |
| 成功 | 成功 | 成功 | 成功 | 允许重新执行三次准入验证，不代表已批准生产集成 |

每次只能提出一个根因假设，并用最小变更验证。若某次修改没有验证假设，应回到证据收集阶段，不在其上叠加其他修复。额度或频率错误只能等待限制恢复；不得通过并发、会话复用或密集重试绕过平台限制。

## 8. 测试先行与验收

### 8.1 纯函数测试

在实现诊断纯函数前先写失败测试，至少覆盖：

- 错误名称和短错误码的安全提取；
- URL 查询参数、Bearer 值、token/secret/password 键值和长随机串脱敏；
- 换行与控制字符清除；
- 超长错误信息截断；
- 字符串、数字、对象等非 `Error` 异常的固定兜底；
- 输出对象不包含堆栈、`cause` 或任意附加属性。

### 8.2 启动流程测试

通过依赖注入的模拟 Browser Binding 和浏览器资源，先写失败测试覆盖：

- 每个阶段成功时进入下一阶段；
- 浏览器启动、上下文创建和页面创建分别失败时返回正确阶段；
- 已创建的页面、上下文和浏览器总会按逆序关闭；
- 清理失败不会覆盖原始诊断结论；
- 成功结果不携带错误字段。

### 8.3 HTTP 边界测试

验证诊断入口只接受约定方法和路径、设置 `no-store`、只返回白名单字段，并拒绝由请求参数指定任意目标 URL。任天堂目标地址仍由探针内部常量控制，防止临时 Worker 被误用为开放浏览器代理。

### 8.4 执行验收

完成单元测试和 TypeScript 类型检查后，严格按本地启动、远程启动、普通导航、任天堂单次提取、任天堂三次提取的顺序执行。每个阶段必须保存开始时间、允许字段和结论；未通过阶段之后的步骤不得执行。

## 9. 文档与提交边界

- 诊断实现只存在于临时目录，不提交临时代码、依赖、日志或结果 JSON。
- 项目仓库只更新本规格、实施计划、文档索引和最终质量结论。
- 任何文档提交前仍须向管理员列出变更范围并取得明确确认，随后在同一操作中提交并推送到 `origin/main`。
- 本规格批准的是根因诊断，不批准临时 Worker 部署、生产代码修改或生产发布。

## 11. 实际诊断结果

1. 本地诊断稳定复现 `browser-launch` / `TypeError` / `Cannot read properties of null (reading 'accept')`。依赖源码对比确认 `@cloudflare/playwright` 1.3.0 已使用新标准 CDP 路径，而 Wrangler 4.31.0 的 Miniflare 仍只实现旧连接路径。
2. 临时 Wrangler 升级到与现有 Workers Types 兼容的 4.107.1，兼容日期调整为运行时支持的 `2026-07-09`。浏览器下载安装完成后，本地 `launch-only` 成功。
3. 远程 `launch-only` 和固定普通测试页导航依次成功，证明账户、远程 Browser Binding、CDP 会话和基础网络均可用。
4. 任天堂单次验证首次因官方目标 URL 没有末尾斜杠而被严格规则误拒绝。经测试先行把零个或一个末尾斜杠规范化为统一 URL 后，单次验证和三次独立复验全部成功。
5. 三次结果的 URL 完全一致，均为 `https://store-jp.nintendo.com/item/software/D70050000064985/`；临时远程预览已关闭且没有形成持久 Worker。

诊断目标已经达到。下一阶段只能是单独的生产集成设计；本规格没有授权修改生产应用或部署。

## 12. 文档自检

- 本规格没有 `TBD`、`TODO`、占位字段或未决数据来源。
- 隔离诊断、任天堂页面验证和生产集成三个阶段边界明确，互不替代。
- 输出白名单与脱敏规则足以保留常见平台错误证据，同时禁止敏感数据和完整响应泄漏。
- 判断矩阵只允许基于已执行阶段形成结论，不会把浏览器启动失败错误归因到任天堂页面。
- 范围只覆盖单一启动故障的定位和原可行性验证恢复，足够由一个实施计划执行。
